Une confirmation débloque l'accès ; la sécurité se réactive après 1 h d'inactivité (config `AUTH_PASSWORD_INACTIVITY_TIMEOUT`). Prop Inertia `auth.password_recently_confirmed` pour éviter de redemander le mot de passe à chaque chargement de page (ex. Scrapping).
- **Vérification confidentialité GitHub** : [VERIFICATION_CONFIDENTIALITE_GITHUB.md](./VERIFICATION_CONFIDENTIALITE_GITHUB.md) — audit des données personnelles (emails, clés, fichiers sensibles), corrections appliquées (scripts Playwright + neutralisation des références à l’ancien site dans la doc règles), 2 mentions conservées pour l’ancienne adresse du JDR.
- **Scrapping (backend)** : [FINALISATION_SCRAPPING_BACKEND.md](./FINALISATION_SCRAPPING_BACKEND.md) — limites depuis config, initiative monster, import range via runMany, robustesse, tests, doc.
- **Scrapping (config)** : [INVENTAIRE_JSON_ET_MIGRATION_BDD_UI.md](../50-Fonctionnalités/Scrapping/INVENTAIRE_JSON_ET_MIGRATION_BDD_UI.md) — inventaire des JSON (rôle, clés utilisées), évaluation transfert BDD + UI admin, plan de migration en 3 phases.

- **Admin caractéristiques — Panel Conversion** : select « Fonction de conversion » dans le panneau Conversion (général et par entité), alimenté par `ConversionFunctionRegistry::options()` (création et fiche caractéristique). Option « Aucune » + liste des fonctions enregistrées côté serveur.
- **Système d’effets unifié — Phase 4 (UI admin)** : CRUD admin Sous-effets et Effets (menu Admin : Sous-effets, Effets) ; bouton « Ajouter un degré » sur un effect ; section « Effets » sur les fiches Sort, Item et Consumable (liste des usages, ajout/édition/suppression, aperçu « effet pour niveau X »). Page d’édition Consommable complétée (formulaire + Effets). Voir [PLAN_MISE_EN_OEUVRE_EFFECTS.md](../50-Fonctionnalités/Spell-Effects/PLAN_MISE_EN_OEUVRE_EFFECTS.md).
- **Harmonisation vues Minimal (hover extended)** : [smoke-test-minimal-views-2026-03-04.md](./smoke-test-minimal-views-2026-03-04.md) — Smoke test visuel frontend sur les vues Minimal des entités item, spell, monster, resource. Validation de l'ordre des champs (métier en haut, techniques en bas) après harmonisation. 15 composants analysés, 0 anomalie détectée. Résumé : [smoke-test-minimal-views-summary.md](./smoke-test-minimal-views-summary.md).
- **Export RGPD « Indisponible »** : Correction du problème où l'export restait en « pending » sans date d'expiration tant qu'aucun worker de queue ne traitait le job. Ajout de l'option `PRIVACY_EXPORT_SYNC=true` pour exécuter l'export immédiatement (sans worker), config `config/privacy.php`. Doc mise à jour ([Privacy-RGPD/README.md](../50-Fonctionnalités/Privacy-RGPD/README.md)).
- **Suppression de compte RGPD — délai de rétractation et admin** : (1) Délai de rétractation (7 jours par défaut, configurable via `PRIVACY_ERASURE_WITHDRAWAL_DAYS`) : la demande n'exécute plus le job immédiatement ; la commande `privacy:process-deletion-requests` (planifiée à 02:00) traite les demandes dont le délai est passé. (2) Bannière « Récupérer mon compte » pour les utilisateurs connectés avec une demande en cours. (3) Message explicite après déconnexion et à la tentative de connexion d'un compte supprimé. (4) Admin : boutons Restaurer et Supprimer définitivement (liste + fiche utilisateur) pour les super_admins.
- **Éléments multiples (Spell & Capability)** : Constante numérique 0-29 (Neutre, Terre, Feu, Air, Eau et combinaisons), migration Capability (element string → integer), `ElementConstants` partagé, composant `ElementDisplay` avec dégradés SCSS et icônes (`storage/app/public/images/icons/caracteristics/`). Voir [PLAN_ELEMENTS_MULTIPLES.md](../50-Fonctionnalités/Elements/PLAN_ELEMENTS_MULTIPLES.md).
- **Scrapping — corrections recherche (2026-03-09)** : (1) **Panoplies** : `$populate=items` sur l’endpoint item-sets pour inclure les équipements dans la liste ; formatter `extractItemIds` gère désormais les items sous forme d’IDs directs. (2) **Ressources** : `$populate=false` + suppression serveur de `recipesThatUse` (recettes utilisant la ressource) — ces recettes sont obtenues à l’inverse via équipements/consommables. (3) **Classes** : utilisation de `breedSpellsId` de l’API breeds (toutes les variantes) en priorité, fallback spell-levels si breedSpellsId vide ; variantes via `GET /spell-variants?breedId=X` (44 sorts base+variantes). (4) **Sorts invocations** : extraction du monstre invoqué depuis les effets spell-levels (effectId 181 = Invoquer, diceNum = ID monstre DofusDB).
- **Scrapping — pipeline import UX et fiabilité (2026-03-09)** : (1) **Progression visuelle** : fallback automatique en mode synchrone si création de job > 15 s (ex. `QUEUE_CONNECTION=sync`) ou si le job reste en `queued` > 60 s (worker non lancé) ; message explicite « mode synchrone, peut prendre plusieurs minutes » pendant l'attente. (2) **Imports multiples** : guard au clic (ne pas relancer si import en cours) ; annulation du batch précédent avant d'en lancer un nouveau. (3) **Normalisation** : la réponse du fallback synchrone est convertie au format attendu (`run_id`, `summary`, `results`, `debug`). (4) Doc `SCRAPPING_ASYNC_JOBS.md` : prérequis `queue:work` + `QUEUE_CONNECTION=database`.
- **Scrapping — option respect_auto_update (2026-03-09)** : Nouvelle option d'import `respect_auto_update` (défaut `true`). Quand activée, seules les entités déjà en BDD avec `auto_update=true` sont mises à jour par les données de l'API ; celles avec `auto_update=false` sont conservées telles quelles (non remplacées). Option exposée dans le panneau « Options & historique » du dashboard scrapping et transmise aux endpoints batch/jobs.
- **Scrapping — option skip_existing (2026-03-10)** : Nouvelle option `skip_existing` pour ne pas appeler l'API ni faire de calcul sur les entités déjà en BDD qu'on ne remplacerait pas. Accélère fortement les imports en masse.
- **Scrapping — mode de mise à jour unifié (2026-03-10)** : Un seul sélecteur `update_mode` remplace les 3 options (replace_mode, respect_auto_update, skip_existing) pour éviter les contradictions. 4 choix : ignorer si existant, mettre à jour si (brouillon/raw) ET auto_update, mettre à jour tous les auto_update, forcer les mises à jour.
- **Scrapping — fiabilité jobs (2026-03-10)** : (1) `DB::reconnect()` au début de chaque job pour éviter "MySQL server has gone away" après un job long. (2) Retries (2 tentatives, backoff 30 s) pour erreurs transitoires. (3) SaveProgress tous les 5 entités au lieu de chaque entité. (4) Doc : lancer `php artisan queue:work --timeout=660` (le worker utilise 60 s par défaut, ce qui tuait les jobs vers 1 min).
- **Effets — type de cible (direct / piège / glyphe)** : `target_type` au niveau de l'effet (Effect.target_type). Intégration complète : déduplication par signature (target_type + area inclus), API (EffectResource), admin CRUD (champ select), affichage badge dans EffectUsagesManager. Commande `effects:rebuild-signatures` pour recalculer les signatures existantes. Valeur par défaut : direct.
- **Effets — chips avec icônes éléments (2026-03-10)** : Colonne « Effets » du tableau sorts en chips : icône élément (Eau, Feu, Terre, Air, Neutre), valeur, tooltip avec target_type/zone/duration. API `effect_usages_chips` + `buildEffectUsagesData()`. Duration 0 ou 1 → « Immédiat ».
- **Zones d'impact — constantes et icônes (2026-03-10)** : `App\Support\AreaConstants` (PHP) et `@/Utils/Entity/Areas` (frontend) pour uniformiser les zones (point, line, cross, circle, rect). Icônes dans `icons/areas/`. Affichage zone avec icône : colonne Zone tableau sorts, profil sort, EffectUsagesManager, admin Effets. Spell::AREAS_SHAPE délègue à AreaConstants::SHAPE_ID_MAP.
- **Scrapping — catégorie des sorts (2026-03-10)** : Par défaut, tous les sorts sont importés comme **sorts de monstre** (`Spell::CATEGORY_CREATURE`). Seule exception : les sorts ajoutés lors du scrapping des **classes** (breeds) sont marqués en **sorts de classe** (`Spell::CATEGORY_CLASS`). Le hint `spell_category_hint` (Orchestrator → IntegrationService) prioritaire sur la valeur convertie ; défaut CREATURE si absent.
- **Affichage des effets (2026-03-10)** : (1) **Tableau sorts** : nouvelle colonne « Effets » affichant le résumé des effect_usages (nom effet, degré, sous-effets). API table : `effect_usages_summary` chargé via relation `effectUsages.effect.effectSubEffects.subEffect`. (2) **Vue dédiée sous-effets** : page Admin « Sous-effets » (`admin/sub-effects`) listant le référentiel (slug, type, template, nb effets). Lien menu Admin à côté d’« Effets ».
- **Scrapping — recherche par niveau monstres (2026-03-09)** : Le filtre `levelMin` / `levelMax` ne fonctionnait pas pour les monstres car l’API DofusDB stocke le niveau dans `grades.0.level` et non en champ racine. Correction : option `filters.levelPath` dans la config d’entité (ex. `grades.0.level` pour monster). Items et panoplies utilisent `level` par défaut.
- **Images scrapping — stockage MEDIA_PATH (2026-03)** : Correction du `ModelAwarePathGenerator` : résolution des alias du morphMap (`spell`, `item`, etc.) pour utiliser les dossiers MEDIA_PATH des modèles (`images/entity/spells/`, `images/entity/items/`, etc.). `EnsureDirectoryMediaFilesystem` crée explicitement les dossiers avant écriture (évite les échecs si dossiers absents). Les nouvelles images scrapping sont stockées au bon endroit. Commande `media:fix-storage-paths` pour migrer les médias existants stockés à la racine. Dépannage : `config:clear`, `queue:restart` (voir SPATIE_MEDIA_LIBRARY.md).
- **Commandes project:init et project:update (2026-03-09)** : (1) **project:init** : initialisation complète du projet (migrations, seeders, types, scrapping ordonné par entité). Vide la queue avant scrapping (sauf `--skip-clear-queue`). Compatible exécution longue (set_time_limit(0), DB::reconnect). Notifie admin/super_admin à la fin (succès, durée, heure). Options : `--fresh`, `--skip-migrate`, `--skip-seeders`, `--skip-scrapping`, `--skip-capabilities`, `--skip-types`, `--noimage`, `--skip-cache`, `--entity=`, `--max-items=`, `--simulate`, `--init-scheduler`, `--skip-clear-queue`, `--skip-notify`. (2) **project:update** : idem (queue, long run, notification). Options : `--entity=`, `--noimage`, `--skip-cache`, `--dry-run`, `--skip-clear-queue`, `--skip-notify`. (3) **Scheduler** : `PROJECT_UPDATE_CRON` (défaut `0 1 1 * *`). (4) **Notification** : type `project_maintenance` (config notifications), canaux par défaut in-app.
- **Pagination côté serveur — tableaux (2026-03-10)** : Nouveau mode `server-side` pour les tableaux d'entités avec gros volumes (10k+ lignes). Au lieu de charger 5000 lignes puis filtrer/trier côté client, l'API reçoit `page`, `limit`, `sort`, `order`, `search`, `filters[key]` et renvoie la page courante + `meta.pagination`. Sorts activés en premier. Voir [TANSTACK_TABLE.md](../30-UI/TANSTACK_TABLE.md).
